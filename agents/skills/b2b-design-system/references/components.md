# Components

Use this file when building or revising hero sections, proof surfaces, offers, CTA blocks, or page architecture.

## Conversion Hierarchy

Every section has a conversion job:

1. Hero
2. Proof
3. Mechanism
4. Offer
5. Objection handling
6. Final CTA

Do not start with visual polish if the hierarchy is still wrong.

## CRO Rules

- Use one primary CTA per page
- Keep the CTA as the highest-contrast action on screen
- Keep headline, sub-headline, primary CTA, and one trust signal above the fold
- Keep social proof within one scroll
- Keep first-contact forms to three fields unless the use case truly needs more
- Use visual flow to guide the eye toward the CTA
- Give the primary CTA at least `48px` of breathing room
- On mobile, use a sticky CTA after the hero when the page is action-oriented

## Hero

Hero job:

- capture attention in `5` to `8` seconds
- state who the offer is for and what changes
- make the next step obvious

Must include:

- one `H1`
- one supporting paragraph
- one primary CTA
- one trust signal above the fold
- one nearby proof or diagnostic surface

Patterns:

- Prefer asymmetric desktop layouts: copy on one side, proof or mechanism on the other
- A centered hero is acceptable only when line length stays tight and the screen still has a strong focal point
- Keep the headline visually dominant and the CTA isolated enough to feel important

## Proof

Place proof before the user scrolls twice:

- metrics with context
- case-study teaser with one concrete outcome
- proof bullets tied to the offer
- logos only if they are treated as part of a designed module

Avoid:

- unlabeled numbers
- flat "trusted by" strips
- empty testimonial theater

## Mechanism

Explain the system in three steps when possible:

1. diagnosis
2. blueprint or prioritization
3. implementation or optimization

If the economic upgrade path is the middle step, make it visually strongest.

## Offer

The offer section should define what the buyer gets, not dump service categories.

Prefer:

- tiered progression
- outcome framing
- explicit scope boundaries
- qualification language such as "not for you if..."

## Objection Handling

Use FAQ or a qualification section to remove resistance, not to repeat features.

Good topics:

- fit
- timeline
- scope and dependencies
- whether ads or extra tools are required
- what happens after the first engagement

## Final CTA

Repeat the primary action with less explanation and more confidence.

Good support copy:

- one payoff recap
- one reassurance line
- low-friction next-step framing

## Audit Mode

When reviewing an existing page:

1. Score it against the quality gate as `pass`, `partial`, or `fail`
2. Identify the top three fixes by conversion impact first
3. Separate repo changes from manual WordPress-editor work
